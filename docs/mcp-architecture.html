devLog MCP Feature

Let external agents update their owner’s projects safely.

Architecture for an MCP server that exposes devLog tools like project lookup, timeline reading, and log creation to Claude, Cursor, local agents, or custom automations.

MCP ServerSupabaseScoped Agent TokensOwner-only Access
Problem

Agents need a controlled bridge into devLog.

What users want

  • “Add today’s progress to my project.”
  • “Summarize this git work and create a devLog entry.”
  • “Update my project status after deployment.”

What we must protect

  • No browser-side service keys.
  • No access to other users’ projects.
  • No broad database writes.
  • Revocable and auditable access.
High-level architecture

The agent never talks directly to Supabase.

External AgentClaude Desktop, Cursor, custom local agent
→
devLog MCP ServerValidates token, exposes safe tools
→
SupabasePostgres, Auth, Storage, RLS
→
devLog AppUser sees updated projects/logs

The MCP server is the permission boundary. It translates agent tool calls into carefully validated Supabase operations.

Required pieces

What we need to add.

1. Token table

Stores hashed agent tokens, owner, scopes, project restrictions, expiry, revoke state.

2. Token UI

Page/modal in devLog where user creates, names, copies, and revokes agent tokens.

3. MCP package

A Node/TypeScript server implementing tools for projects and logs.

4. Authorization layer

Every call checks token validity, scopes, owner_id, and allowed_project_ids.

5. Audit log

Record what agent did, when, and against which project/log.

6. Deployment

Run locally for personal agents or deploy as a hosted MCP/HTTP bridge later.

Database additions

Agent tokens are scoped, hashed, and revocable.

create table public.agent_tokens (
  id uuid primary key default uuid_generate_v4(),
  owner_id uuid not null references public.profiles(id) on delete cascade,
  name text not null,
  token_hash text not null unique,
  scopes text[] not null default '{}',
  allowed_project_ids uuid[], -- null = all owner projects, or restrict list
  expires_at timestamptz,
  revoked_at timestamptz,
  last_used_at timestamptz,
  created_at timestamptz not null default now()
);

create table public.agent_audit_logs (
  id uuid primary key default uuid_generate_v4(),
  token_id uuid references public.agent_tokens(id) on delete set null,
  owner_id uuid not null references public.profiles(id) on delete cascade,
  action text not null,
  project_id uuid,
  log_id uuid,
  metadata jsonb not null default '{}',
  created_at timestamptz not null default now()
);
Permission model

Each MCP tool requires a scope.

ToolRequired scopeExtra check
devlog_list_projectsread_projectsOnly owner’s projects
devlog_get_project_timelineread_logsProject belongs to owner / allowed list
devlog_create_logcreate_logProject belongs to owner / allowed list
devlog_update_logupdate_logLog’s project belongs to owner
devlog_update_projectupdate_projectProject belongs to owner

Rule: agent can only mutate projects owned by the token owner.

How it works

Example: “Add today’s progress log.”

User
Creates token in devLog: name “Claude Desktop”, scopes: read_projects + create_log.
Agent
User configures MCP with DEVLOG_AGENT_TOKEN.
Agent calls devlog_list_projects to find project.
Agent calls devlog_create_log with title/content/mood/media metadata.
MCP
Hashes provided token and finds active token row.
Checks scope and project ownership.
Writes log to Supabase and records audit event.
devLog
New timeline log appears in the user’s project.
MCP server internals

Suggested repo structure.

mcp/
  package.json
  tsconfig.json
  src/
    index.ts                  # start MCP stdio server
    supabase.ts               # service-role client, server-only
    auth.ts                   # validate token + check scopes
    audit.ts                  # write audit log
    tools/
      projects.ts             # list/get/update project tools
      logs.ts                 # timeline/create/update log tools
    schemas.ts                # zod input schemas

The MCP server can run locally through stdio first. Later it can be deployed as a hosted HTTP/SSE MCP server if needed.

Security design

Use service-role carefully, then enforce app-level auth.

Do

  • Keep service role only in MCP server env.
  • Hash tokens with SHA-256 or bcrypt.
  • Check owner on every project/log write.
  • Use scopes for each tool.
  • Record audit logs.
  • Allow revocation and expiry.

Do not

  • Expose service role in React/Vite.
  • Let agents pass arbitrary SQL.
  • Trust project_id without ownership check.
  • Store raw tokens in database.
  • Give agents full user JWT access.
Tool contract

Minimal MVP tools.

devlog_list_projects()
→ [{ id, title, description, visibility, updated_at }]

devlog_get_project_timeline({ project_id })
→ { project, logs: [...] }

devlog_create_log({
  project_id,
  title,
  content,
  mood?: "building" | "shipped" | "stuck" | "reflecting" | "inspired" | "learning",
  visibility?: "private" | "public" | "unlisted"
})
→ created log

devlog_update_project({ project_id, title?, description?, tags?, visibility? })
→ updated project
Agent configuration

How a user connects an agent.

{
  "mcpServers": {
    "devlog": {
      "command": "node",
      "args": ["/path/to/devLog/mcp/dist/index.js"],
      "env": {
        "DEVLOG_SUPABASE_URL": "https://...supabase.co",
        "DEVLOG_SUPABASE_SERVICE_ROLE_KEY": "server-only-key",
        "DEVLOG_AGENT_TOKEN": "dl_agent_xxxxxxxxx"
      }
    }
  }
}

Note: for local MVP, the user/developer runs the MCP server. For hosted MCP, the service role stays on your server and users only provide agent token.

Implementation phases

Recommended roadmap.

Phase 1 — Local MVP

  • Add tables.
  • Add MCP package.
  • Hardcode/manual create token initially.
  • Tools: list projects, create log.

Phase 2 — Productized

  • Token management UI.
  • Audit log UI.
  • More tools: update project/log, media metadata.
  • Better docs and examples.

Phase 3 — Hosted

  • Remote MCP endpoint.
  • OAuth/user-managed connections.
  • Rate limits.
  • Team/collaborator permissions.

Phase 4 — Intelligent workflows

  • Git commit summaries.
  • Release-note to log automation.
  • Daily project journal assistant.
Final architecture

Simple rule: MCP acts as a secure project-writing adapter.

External agents ask to perform devLog actions.

The MCP server validates an owner-scoped token, checks tool scopes, verifies project ownership, writes to Supabase, and records an audit event.

devLog users keep control through token scopes, expiry, revocation, and audit history.

This gives agents useful automation without giving them unsafe database access.
